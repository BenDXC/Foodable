## ü§ù Contributing to Documentation

When adding documentation:
1. Use clear, concise language
2. Include code examples where helpful
3. Keep formatting consistent
4. Update the index when adding new docs
5. Test all commands and examples

## üìù Documentation Standards

### Markdown Style
- Use ATX-style headers (`#` not `===`)
- Include code blocks with language tags
- Use tables for structured data
- Add emojis for visual navigation (sparingly)

### Code Examples
- Always test before documenting
- Include both input and expected output
- Show error cases when relevant
- Use realistic examples

### API Documentation
- Document all endpoints
- Include request/response examples
- List all parameters
- Show error responses

## üîÑ Keeping Docs Updated

Documentation should be updated when:

- Changing APIs or interfaces
- Modifying configuration
- Fixing bugs that affect usage
- Refactoring architecture
